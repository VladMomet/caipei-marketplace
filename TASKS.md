# TASKS — что доделать перед запуском

Это передача в работу разработчику. Архитектура и бэкенд работают;
ниже — задачи, которые нужно закрыть для нормального запуска.

Помечено приоритетом: **MUST** — нельзя без этого запускаться,
**SHOULD** — желательно к запуску, **NICE** — после первых заказов.

---

## MUST — до запуска

### 1. ✅ Реквизиты юр.лица — заполнены

Стоят корректные данные ИП Оболенский Владимир (ИНН, ОГРНИП, адрес, email)
в `src/lib/constants.ts` → `LEGAL_ENTITY`.

Что осталось: при необходимости актуализировать `phone` (сейчас плейсхолдер
`+7 (495) 000-00-00`) — менеджерский номер.

### 2. Юридическая обвязка

- Зарегистрироваться в РКН как оператор ПДн (бесплатно, через Госуслуги).
- Согласовать тексты `/legal/privacy`, `/legal/offer`, `/legal/terms`
  с юристом — структура есть, но конкретные реквизиты/сроки нужны ваши.
- Открыть валютный счёт для расчётов с китайскими поставщиками
  (если ещё не открыт).
- **152-ФЗ и расположение БД** — при деплое на Yandex Cloud (Москва,
  ru-central1-a) требование ст. 18 ч. 5 закрывается автоматически.
  Дополнительных согласий на трансграничную передачу не нужно.

### 3. Заменить тестовый Telegram-токен

В репозитории / `.env.example` стоит тестовый токен бота, который был
передан через переписку. После первого деплоя:

1. Открыть [@BotFather](https://t.me/BotFather) → `/revoke` → выбрать бота
   → получить новый токен.
2. Обновить `TELEGRAM_BOT_TOKEN` в `.env.local` **на VM** во всех трёх
   инстансах (`/opt/caipei/caipei-*/.env.local`).
3. `pm2 restart all`.
4. **Перерегистрировать webhook** с новым токеном (см. DEPLOY.md шаг 10).

### 4. Прогон полного флоу через тестовую регистрацию

Тестовый сценарий:

1. Открыть сайт в incognito.
2. Открыть каталог, добавить 2 товара в корзину со шагом ±10.
3. Перейти в `/checkout` → форма попросит зарегистрироваться.
4. Зарегистрироваться как физлицо.
5. Подтвердить заказ → менеджеры получают 🟢 сообщение в Telegram.
6. Менеджер нажимает «В работу» → статус в `/account/orders` меняется,
   сообщение в Telegram редактируется.
7. Открыть `/sourcing`, отправить заявку с 2 фото → 🟡 в Telegram +
   2 отдельных сообщения с фото-референсами.

### 5. Купить домен и SSL

Через регистратор RU-Center / reg.ru / любой другой.

## SHOULD — к запуску или сразу после

### 6. CNY → RUB курс

Сейчас в `src/lib/constants.ts` константа `CNY_TO_RUB = 11`. Это значение
показывается в тикере. При сильных колебаниях курса надо обновлять.

**Решение 1** — менять руками раз в неделю в `constants.ts` и деплоить.
**Решение 2** — таблица `settings` в БД + cron-задача (Vercel Cron / Yandex
Cloud Function), которая берёт курс с api.cbr-xml-daily.ru/daily_json.js.

### 7. Загрузка каталога — расширить

Сейчас импортёр работает только с `nap.xlsx` (бренд JewelryMeverly). Если
будете добавлять другие бренды:

- В `BRAND_FILTER` в `scripts/import-products.ts` либо убрать фильтр,
  либо передавать через ENV.
- Проверить `CATEGORY_RULES` — может понадобиться расширить набор
  ключевых слов под новый ассортимент.
- Поле `material` в текущем датасете пустое. Если в новых файлах будет
  колонка с материалом — добавить парсинг в `parseRow`.

### 8. Фото товаров

Сейчас фото грузятся напрямую с `cbu*.alicdn.com` через наш `img-proxy`.
Это работает, но имеет два риска:

- Картинка с alicdn может пропасть, если поставщик удалит товар.
- При большом трафике 1688 может начать блокировать наш proxy.

**Решение**: сделать одноразовую заливку всех фоток в Yandex Object
Storage или Selectel S3, и в `productPhotos.url` записывать наш URL.
Скрипт `scripts/cache-photos.ts` — короткий: проход по `productPhotos`,
скачать через прокси, залить в S3, обновить URL.

### 9. Email-уведомления клиенту (опционально)

Сейчас связь с клиентом по факту заказа — телефонный звонок менеджера.
Это нормально для b2b с маленьким потоком. Но email при смене статуса
(«Ваш заказ № CP-XXXXXX в работе») добавит профессиональности.

- Использовать Mailgun / Sendpulse (есть бесплатный тариф).
- Добавить ENV `SMTP_*` и `EMAIL_FROM`.
- Триггер: в webhook'е, после обновления статуса заказа, если у
  пользователя есть email — отправить шаблонное письмо.

### 10. Защита от спама

Сейчас регистрация и форма подбора открыты для всех. Добавить:

- Rate-limit на `/api/auth/register` (10 регистраций / IP / час) — через
  `@upstash/ratelimit` или простой memory-limiter.
- Капча Yandex SmartCaptcha на форме подбора (бесплатна).

### 11. SEO

- Sitemap.xml — добавить `src/app/sitemap.ts`.
- robots.txt — добавить `src/app/robots.ts`.
- JSON-LD на странице товара уже есть в metadata, но проверить через
  валидатор schema.org.
- OG-image на главной — статический баннер 1200×630 с логотипом.

## NICE — после первых заказов

### 12. Свой авто-комплит города

Сейчас `/cities` показывает 10 захардкоженных городов. Можно
интегрировать DaDaTa.ru для подсказок при вводе любого адреса.

### 13. Telegram bot — расширенные команды

В дополнение к кнопкам статусов добавить команды:

- `/orders` — список активных заказов
- `/order CP-XXXXXX` — детали по номеру
- `/picks` — список заявок на подбор
- `/stats` — сводка за день

Это пишется отдельным polling-обработчиком (или расширением webhook'а).

### 14. Интеграция с CRM

Когда команда менеджеров вырастет за 3 человека — захочется amoCRM или
Bitrix24. Хук в `/api/orders/route.ts` и `/api/sourcing-requests/route.ts`
параллельно с Telegram-нотификацией создаёт сделку в CRM.

### 15. Аналитика

- Яндекс.Метрика — есть переменная `NEXT_PUBLIC_YANDEX_METRICA_ID` в
  `.env.example`, надо вставить в `<head>` и проверить отправку событий.
- Воронка: visit → cart → register → order.
- Atribution по UTM.

### 16. Документы ВЭД из шаблона

Сейчас при подтверждении заказа менеджер вручную готовит контракт и
инвойс. Сделать генератор PDF из шаблона по данным заказа — сэкономит
час на каждый заказ.

## Известные ограничения текущего MVP

- **MOQ** — 10 шт на позицию. Если клиент хочет 5 — менеджер пишет
  ему лично, корзина так не оформится.
- **Оплата** — только через менеджера (счёт от ИП/ООО). Онлайн-оплаты
  нет, чтобы не подключать ОФД и онлайн-кассу (54-ФЗ).
- **Восстановление пароля** — только через менеджера. Email-флоу
  «забыли пароль» не реализован осознанно (email опционален).
- **Аудит-лог** — нет таблицы для логирования действий менеджеров
  (решение MVP — менеджеров мало, споров не ожидается).
- **Лимит Telegram** — длинные заказы (20+ позиций) могут не уместиться
  в 4096 символов одного сообщения. Сейчас риск низкий (типовой заказ
  3-5 позиций); если станет проблемой, добавить разбивку.
